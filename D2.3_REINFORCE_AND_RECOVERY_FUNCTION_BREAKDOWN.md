### Audit findings / MVP contract note
Observed repository reality:
- `docs/LESSON_STATE_MACHINE.md` already defines `reinforce` as a first-class session mode and documents the key triggers conceptually: repeated incorrect answers, low confidence, visible confusion, and weak-area patterns.
- `docs/FEEDBACK_TEMPLATE_CATALOG.md` already defines the tone and recovery moves expected once struggle is detected: simplify, narrow scope, use encouragement-first language, and restore momentum with a smaller win.
- `docs/API_ENDPOINT_SPEC.md` currently exposes session CRUD and progress weak-area endpoints, but does **not** yet define any dedicated reinforce/recovery recommendation or summary endpoint.
- Existing backend surface is partial only:
  - `backend/src/sessions/service.js` and controller/repository code already accept `reinforce` as a valid session mode.
  - `backend/src/progress/service.js` already derives project/topic weak areas and a `struggling` progress state from low mastery.
  - No dedicated `reinforce` or `recovery` backend module, controller action, route, or response contract exists yet.

Minimum D2.3 MVP contract for later slices:
- **Struggle detection inputs:** latest active session mode/status, recent answer attempts, and latest progress weak-area snapshot.
- **MVP struggle rule:** learner is considered `struggling` when any of these are true:
  1. latest project/topic progress state is already `struggling`, or
  2. there are at least 2 recent incorrect attempts on the same concept/topic within the active session, or
  3. there is at least 1 recent incorrect attempt plus an existing weak-area hit for the current outline item/topic.
- **MVP recovery actions:** return exactly one primary recommendation from a small deterministic set:
  - `continue` — no recovery needed
  - `review` — recap before more pressure
  - `reinforce` — switch session into supportive recovery mode
  - `fallback_easier_question` — serve a simpler question/worked-example step on the same weak concept
- **MVP response shape target for later implementation:** recommendation payload should include at least `isStruggling`, `reason`, `recommendedAction`, `currentMode`, `targetOutlineItemId`, and optional `supportMessage` aligned with the feedback catalog.
- **Session-close summary target:** if struggle was detected during the session, end-of-session summary should name 1 weak area, 1 recovery action taken or recommended, and 1 next-step suggestion in non-shaming language.
