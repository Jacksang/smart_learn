## D3.1.1 — Audit ingestion strategy, schema, and current backend surface
- [x] Verify the existing ingestion strategy and current material/file-related backend surface

### D3.1.1 audit note — verified MVP ingestion contract

#### Verified current state
- `docs/FILE_INGESTION_STRATEGY.md` already defines the intended MVP ingestion direction: support plain text, PDF, DOCX, and image OCR; normalize everything into extracted text plus traceable metadata.
- `docs/API_ENDPOINT_SPEC.md` already reserves `POST /api/projects/:projectId/materials/upload`, but only at a high level. It does not yet define extraction-state fields or the exact synchronous MVP response contract.
- `backend/db/schema/001_baseline.sql` already gives `source_materials` a usable first-pass persistence surface for ingestion: `source_kind`, `material_type`, `title`, `original_file_name`, `mime_type`, `storage_path`, `raw_text`, `extracted_text`, `weight`, `is_active`, and `source_version`.
- Current backend materials code already supports creating and updating `source_materials` records from JSON payloads, including file-related metadata fields, and already triggers outline refresh after material create/update.
- Current backend does **not** yet implement a real ingestion pipeline: no ingestion module, no extraction services, no extraction status/failure metadata persistence, and no upload controller/route wired to `source_materials`.
- `backend/utils/fileUpload.js` exists but is only scaffolding right now: it stores files under `uploads/outlines`, allows PDF/TXT/DOCX only, excludes images, and is not yet the actual project-material ingestion contract.

#### Confirmed existing persistence vs missing pieces
Already present for MVP start:
- project-linked `source_materials` persistence
- raw text persistence for pasted/manual text
- file metadata persistence for uploaded artifacts
- user/system weighting and active-state handling

Still missing before full D3.1 completion:
- persisted extraction lifecycle fields such as `status`, `failure_reason`, `extractor_type`, `quality_signal`, `file_size_bytes`, or checksum
- dedicated stored-file helper for project materials
- PDF extraction service
- DOCX extraction service
- image OCR service
- ingestion orchestrator that turns upload -> extracted text -> material record update
- material upload endpoint/controller behavior that uses the ingestion pipeline instead of raw JSON create

#### Minimum MVP ingestion contract locked for implementation
D3.1 implementation should treat the MVP contract as:
- Supported upload types: `application/pdf`, DOCX, and image uploads (`png`, `jpg`, `jpeg`, `webp` if stack support is straightforward). Plain pasted text continues through the existing non-upload materials flow.
- Storage model: save the original uploaded file first, then create/update one `source_materials` row for the project that holds the canonical file metadata plus normalized extracted text.
- Material linkage: one uploaded file maps to one `source_materials` record in MVP.
- Text fields: `raw_text` remains for direct text submissions; uploaded-file ingestion should primarily populate `extracted_text`.
- Initial route contract: `POST /api/projects/:projectId/materials/upload` accepts multipart `file` plus optional `title` and `weight`.
- Initial response contract: create the material record even if extraction is partial/failed, as long as file storage succeeded; return the material plus ingestion result metadata from the controller layer.
- Failure rule: upload/storage failure blocks record creation; extraction failure does not delete the stored file reference and must remain machine-detectable for retry logic.
- Schema constraint for downstream slices: if new extraction metadata is required, add it in a dedicated migration or bounded schema change rather than changing the above contract ad hoc during later implementation slices.

### Business logic
Before implementing file ingestion, the system needs one clear contract for which file types are supported, how files are stored, and how extracted text flows into source materials.

### Exact coding problem
Audit current docs/schema/code so later slices can implement file ingestion without guessing about storage locations, extraction outputs, or material-linking behavior.
