  - Missing `backend/src/materials/` runtime module set: at minimum `repository.js`, `controller.js`, and `router.js` for `source_materials` CRUD/list flows bound to a project.
  - Missing route registration in `backend/config/server.js` for `/api/projects` and `/api/materials` (or equivalent project-scoped material endpoints).

  - Missing tests covering PostgreSQL-backed project/material repository behavior and mounted routes; current runtime only exposes users, outline, questions, and answers.
- [x] Record required table/column/API alignment notes
  - `learning_projects` table contract for MVP runtime should expose: `id`, `user_id`, `title`, `description`, `subject`, `status`, `current_mode`, `current_outline_id`, `created_at`, `updated_at`.
  - `learning_projects.current_outline_id` is nullable and points to `outlines.id` with `ON DELETE SET NULL`; project read endpoints should therefore tolerate `currentOutlineId = null` and must not assume an outline already exists.
  - Project list/read queries should be scoped by authenticated `user_id`, matching the existing `protect` middleware pattern already used by outline/question/answer routes.
  - `source_materials` table contract for MVP runtime should expose: `id`, `project_id`, `source_kind`, `material_type`, `title`, `original_file_name`, `mime_type`, `storage_path`, `raw_text`, `extracted_text`, `weight`, `is_active`, `source_version`, `created_at`, `updated_at`.
  - Material writes must respect schema constraints directly: `project_id` required FK, `weight >= 0`, `source_version > 0`, `is_active` default `true`, and default `weight = 1.00` when callers omit weighting.
  - API shape should use project-scoped ownership checks even if materials are mounted at `/api/materials`: every create/read/update/list flow must verify the referenced project belongs to `req.user.id` before returning or mutating rows.
  - Recommended MVP routes from the current Express mounting style: `GET /api/projects`, `POST /api/projects`, `GET /api/projects/:projectId`, `PATCH /api/projects/:projectId`, `GET /api/projects/:projectId/materials`, `POST /api/projects/:projectId/materials`, and `PATCH /api/materials/:materialId` for status/weight/content updates.
  - Response/request field names should stay close to schema names (snake_case in SQL, camelCase mapping only at controller boundary if adopted consistently); avoid repeating the current outline runtime mismatch where `backend/src/outline/repository.js` still queries legacy columns such as `course_title`, `source_type`, `topics`, and `ai_summary` that do not exist in the committed PostgreSQL `outlines` schema.
  - Because `backend/config/server.js` currently mounts only `/api/users`, `/api/outline`, `/api/questions`, and `/api/answers`, adding project/material APIs requires explicit router registration there before any integration tests can pass.
