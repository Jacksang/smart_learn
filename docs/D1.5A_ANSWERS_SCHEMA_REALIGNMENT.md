# D1.5.A Answers Module Schema Realignment

## Goal
Move the existing answers module off the legacy `answers` table assumptions and onto the PostgreSQL MVP shape centered on `answer_attempts` plus project-scoped ownership.

## Runtime/schema gaps confirmed
- `backend/src/answers/repository.js` previously queried a non-existent legacy `answers` table and legacy columns like `submitted_answer`, `feedback`, and `attempt_number`.
- The live PostgreSQL MVP schema defines `answer_attempts` with `user_answer`, `feedback_text`, `attempt_no`, `answered_at`, and project linkage through `project_id`.
- `backend/src/answers/controller.js` previously looked up questions only by `req.body.questionId`, then counted attempts by `(user_id, question_id)` even though current ownership is project-scoped through `learning_projects.user_id` and `questions.project_id`.
- `backend/src/answers/router.js` previously exposed only top-level `/api/answers` routes, while the API spec defines project-scoped answer endpoints under `/api/projects/:projectId/...`.

## D1.5.A code realignment completed
- Replaced the repository surface with PostgreSQL-MVP-aligned helpers:
  - `listByQuestionForProjectAndUser(...)`
  - `listRecentByProjectForUser(...)`
  - `countAttemptsByQuestionInProject(...)`
  - `createAnswerAttempt(...)`
- Added answer-attempt row mapping that preserves schema field names (`user_answer`, `feedback_text`, `attempt_no`, `answered_at`) and joins question context only as read metadata.
- Reworked the controller to use project/question route params and enforce question ownership through `findByIdForProjectAndUser(...)` before returning question history.
- Reworked the router to the API-spec path layout:
  - `GET /api/answers/projects/:projectId/questions/:questionId/answers`
  - `POST /api/answers/projects/:projectId/questions/:questionId/answers`
  - `GET /api/answers/projects/:projectId/answers/history`
  - `POST /api/answers/projects/:projectId/answers/evaluate`

## Intentional D1.5.A boundary
This checkpoint does **not** finish submission/evaluation logic yet.
Those behaviors are left as explicit project-scoped scaffolds returning `501` so D1.5.C and D1.5.D can implement them without carrying forward the legacy table/route shape.
