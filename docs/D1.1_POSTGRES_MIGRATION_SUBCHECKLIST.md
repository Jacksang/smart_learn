## D1.1.C — PostgreSQL bootstrap design
- [x] Define environment variables and connection config structure
  - Defined (2026-04-04): standardize on discrete `DB_*` variables for MVP — `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD` — with optional follow-on support for `DB_SSL`, `DB_POOL_MAX`, `DB_IDLE_TIMEOUT_MS`, and `DB_CONNECTION_TIMEOUT_MS`.
  - Connection structure decision: keep one shared `pg` pool config module that normalizes env input into `{ user, host, database, password, port, ssl?, max?, idleTimeoutMillis?, connectionTimeoutMillis? }` and exports thin helpers (`query`, `pool`, `connect`).
  - Stability note: preserve the current `backend/config/database.js` discrete env mapping instead of introducing `DATABASE_URL` during D1.1; add parsing/SSL/pool tuning later without changing the public config contract.
- [x] Define schema bootstrap / migration approach
  - Decision (2026-04-04): use a SQL-first bootstrap path for D1.1 — committed `.sql` schema files plus a small Node runner that executes them through the existing shared `pg` pool/config.
  - Why: the repo already has `backend/config/database.js` for PostgreSQL access, but no migration framework or migration directory exists yet; plain SQL keeps the schema reviewable and avoids adding another migration variable during Mongo runtime retirement.
  - Execution boundary: bootstrap should run as an explicit setup command, not automatically on normal server startup; the first D1.1 schema file is the baseline bootstrap, with room to add ordered SQL migrations or adopt a formal tool later.
- [x] Define folder/file structure for SQL or migration files
  - Defined (2026-04-04): place PostgreSQL artifacts under `backend/db/`, with committed schema SQL in `backend/db/schema/001_baseline.sql` and the execution helper in `backend/db/scripts/apply-sql-file.js`.
  - Ordering rule: SQL files use numeric prefixes from the start so the baseline can stay `001_baseline.sql` and later lightweight migrations can extend the same sequence if needed.
  - Boundary note: keep runtime config in `backend/config/database.js`, keep executable schema helpers in `backend/db/scripts/`, and do not auto-run `backend/db/schema/*` during normal server startup.
