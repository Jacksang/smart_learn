# D1.2 Runtime and Schema Alignment Audit

Date: 2026-04-04
Scope: `learning_projects` and `source_materials` runtime/schema alignment for the PostgreSQL-backed backend.

## Current-state findings
- The committed PostgreSQL schema already defines `learning_projects` and `source_materials` tables.
- The runtime does not yet include matching feature modules under `backend/src/projects/` or `backend/src/materials/`.
- `backend/config/server.js` currently mounts only `/api/users`, `/api/outline`, `/api/questions`, and `/api/answers`, so project/material routes are not exposed.
- Existing backend patterns suggest new project/material features should follow the repository/controller/router structure already used by other modules.

## Required runtime pieces still missing

### Projects
Need a `backend/src/projects/` module set with, at minimum:
- `repository.js`
- `controller.js`
- `router.js`

Expected MVP operations:
- create project
- list projects for authenticated user
- get single project for authenticated user
- update project fields needed by the MVP

### Materials
Need a `backend/src/materials/` module set with, at minimum:
- `repository.js`
- `controller.js`
- `router.js`

Expected MVP operations:
- create material under a project
- list materials for a project
- update material fields such as status, weight, and text content
- enforce project ownership through the parent project

### Route registration
Need explicit registration in `backend/config/server.js` for:
- `/api/projects`
- project-scoped material routes such as `/api/projects/:projectId/materials`
- any direct material update route such as `/api/materials/:materialId`

### Service/test gaps
- No service-layer logic yet ties project records to material weighting or fallback behavior before outline/question generation.
- No PostgreSQL-backed tests yet cover project/material repositories or their mounted routes.

## Table and API alignment notes

### `learning_projects`
MVP runtime contract should expose:
- `id`
- `user_id`
- `title`
- `description`
- `subject`
- `status`
- `current_mode`
- `current_outline_id`
- `created_at`
- `updated_at`

Important alignment notes:
- `current_outline_id` is nullable and references `outlines.id` with `ON DELETE SET NULL`.
- Read flows must tolerate `currentOutlineId = null`.
- All list/read/update access should be scoped to authenticated `user_id` using the existing `protect` middleware pattern.

### `source_materials`
MVP runtime contract should expose:
- `id`
- `project_id`
- `source_kind`
- `material_type`
- `title`
- `original_file_name`
- `mime_type`
- `storage_path`
- `raw_text`
- `extracted_text`
- `weight`
- `is_active`
- `source_version`
- `created_at`
- `updated_at`

Important alignment notes:
- Writes must preserve schema constraints directly.
- `project_id` is required.
- `weight >= 0`.
- `source_version > 0`.
- `is_active` defaults to `true`.
- `weight` should default to `1.00` when omitted by callers.
- Material endpoints must verify the referenced project belongs to `req.user.id` before returning or mutating rows.

## Recommended MVP routes
- `GET /api/projects`
- `POST /api/projects`
- `GET /api/projects/:projectId`
- `PATCH /api/projects/:projectId`
- `GET /api/projects/:projectId/materials`
- `POST /api/projects/:projectId/materials`
- `PATCH /api/materials/:materialId`

## Key implementation caution
Avoid repeating the existing outline runtime/schema mismatch: `backend/src/outline/repository.js` still queries legacy columns such as `course_title`, `source_type`, `topics`, and `ai_summary`, which do not exist in the committed PostgreSQL `outlines` schema. New project/material code should align to the actual schema from the start.
