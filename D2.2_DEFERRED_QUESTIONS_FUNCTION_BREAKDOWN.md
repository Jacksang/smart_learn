## D2.2.1 — Audit deferred-question schema/spec/current code surface
- [x] Verify schema and API expectations for deferred questions
- [x] Confirm whether any deferred-question backend code already exists
- [x] Define the minimum MVP contract before implementation starts
+
+### D2.2.1 audit note — MVP contract confirmed
+- Schema already includes `deferred_questions` in `backend/db/schema/001_baseline.sql` with: `id`, `project_id`, nullable `session_id`, nullable `outline_item_id`, `question_text`, `defer_reason`, `status` defaulting to `deferred`, nullable `brief_response`, timestamps, and nullable `resolved_at`.
+- Existing schema indexes support MVP list flows by project/session/topic and status; no schema change is required to start D2.2 implementation.
+- API spec already expects project-scoped endpoints for list/create/get/update plus a `revisit` action, but the minimum MVP needed by the next slices is narrower: create, list, and status update.
+- Lesson-state rules confirm deferred questions are meant for off-track or too-deep questions, should preserve current lesson flow, and should be revisited only at natural boundaries.
+- Current backend code surface check: no `backend/src/deferred-questions/*` files exist yet, so there is no repository/service/controller/router implementation to extend.
+
+#### Minimum MVP contract before implementation
+- Ownership boundary: every deferred question must belong to one owned `project_id`; `session_id` and `outline_item_id` are optional context only.
+- Create contract: accept `sessionId?`, `outlineItemId?`, `questionText`, and `deferReason`; persist with initial `status='deferred'`.
+- List contract: return deferred questions for one owned project, stably ordered by `created_at DESC` unless a more specific session filter is added later.
+- Update contract: allow status updates for one owned deferred question; MVP statuses should be limited to `deferred`, `revisited`, and `resolved` because those are the smallest set consistent with schema default plus revisit flow.
+- Response contract: return the persisted deferred-question resource using existing API envelope conventions and `camelCase` JSON fields.
+- Out-of-scope for MVP start: dedicated priority/revisit-trigger/source-state fields, automatic queue ordering beyond stable created-time ordering, and separate revisit orchestration beyond marking a question as revisited.
