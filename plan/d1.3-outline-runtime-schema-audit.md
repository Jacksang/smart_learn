# D1.3 Outline Runtime/Schema Alignment Audit

## Scope
Audit of the current outline implementation against the committed PostgreSQL MVP schema.

## Files inspected
- `backend/db/schema/001_baseline.sql`
- `backend/src/outline/repository.js`
- `backend/src/outline/controller.js`
- `backend/src/outline/router.js`
- `backend/src/outline/parser.js`
- `backend/src/projects/repository.js`
- `backend/src/materials/repository.js`
- `backend/models/Outline.js`

## Confirmed schema shape
### `outlines`
Expected columns:
- `id`
- `project_id`
- `version_no`
- `generation_source`
- `status`
- `summary`
- `created_from_material_version`
- `created_at`
- `updated_at`

### `outline_items`
Expected columns:
- `id`
- `outline_id`
- `parent_item_id`
- `title`
- `summary`
- `key_points`
- `sort_order`
- `depth_level`
- `mastery_state`
- `created_at`
- `updated_at`

## Runtime/schema mismatches
1. `backend/src/outline/repository.js` still targets a legacy blob/document model with fields like `user_id`, `course_title`, `subject`, `source_type`, `source_path`, `topics`, and `ai_summary`, which do not exist in the PostgreSQL schema.
2. Runtime scoping is still user-direct (`req.user.id` / `userId`) instead of project-based via `learning_projects.user_id` and `outlines.project_id`.
3. Topic hierarchy is still treated as inline `topics` JSON, but the schema requires normalized `outline_items` rows.
4. Existing controller/router behavior only supports legacy list/create/upload flows; it does not support project-scoped versioned outline creation, current-outline tracking, or refresh behavior.
5. `parser.js` still emits legacy topic fields (`description`, `learningObjectives`, `difficulty`) that do not map to `outline_items`.

## Required alignment decisions
1. Repository methods must read/write only the committed relational columns for `outlines` and `outline_items`.
2. Authorization and listing must flow through project ownership checks.
3. New outline creation must assign per-project `version_no` and update `learning_projects.current_outline_id` when appropriate.
4. Generation flows must populate `created_from_material_version` from source materials.
5. Outline trees must be persisted to `outline_items` with explicit parent/child ordering and `depth_level`.
6. API payloads should return outline metadata plus normalized or reconstructed items, not raw legacy `topics` blobs.
7. Any parser output must be mapped into schema-backed fields and drop unsupported legacy fields unless the schema is intentionally extended later.

## Outcome
The audit confirms D1.3 implementation should replace the remaining legacy outline document contract with a project-scoped PostgreSQL persistence layer centered on `outlines` and `outline_items`.
