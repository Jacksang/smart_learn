Progress snapshots should be derived from persisted learning activity, not ad-hoc frontend math. The backend needs one repository layer that turns `answer_attempts` plus `questions` plus `outline_items` into deterministic aggregate inputs.

### Exact coding problem

- Do not bury all business heuristics inside one opaque query if plain JS service logic would be clearer

## D1.5.E3 — Add progress service for deterministic snapshot calculation
- [ ] Add a service that turns aggregate inputs into project/topic snapshot payloads
- [ ] Implement explainable MVP formulas for `completion_percent`, `mastery_score`, and `progress_state`
- [ ] Implement deterministic `weak_areas`, `strength_areas`, and `summary_text` generation
- [ ] Add focused unit tests for zero/low/mid/high mastery scenarios

### Business logic
The app needs a simple, explainable way to say how far a learner has progressed, how strong their mastery is, and which topics are weak or strong. This must be deterministic so later retrieval endpoints can trust persisted results.

### Exact coding problem
Implement a pure calculation layer that accepts aggregate metrics and emits snapshot data ready to persist.

### Required function behavior
Suggested functions:
- `buildProjectProgressSnapshot(aggregate)`
- `buildTopicProgressSnapshots(topicAggregates)`
- `deriveProgressState({ completionPercent, masteryScore })`
- `deriveWeakAndStrengthAreas(topicSnapshots)`
- `buildProgressSummary({ projectSnapshot, weakAreas, strengthAreas })`

### Suggested MVP formulas
Workers should keep formulas simple and explicit unless the schema/spec says otherwise:
- `completion_percent`:
  - based on answered unique questions / total questions in scope
- `mastery_score`:
  - based primarily on average score, optionally tempered by recent correctness
- `progress_state`:
  - deterministic buckets such as `not_started`, `struggling`, `in_progress`, `strong`, `mastered`
- `weak_areas`:
  - topics with low mastery and enough signal to be meaningful
- `strength_areas`:
  - topics with high mastery and enough signal to be meaningful
- `summary_text`:
  - short human-readable explanation of project status + key weak/strong themes

### Weak-area logic expectations
A topic should be considered weak only by explainable rules, for example:
1. it has attempts or answered questions
2. its mastery score is below a chosen threshold
3. optionally its recent correctness trend is poor if recent-history data is available

Avoid black-box weighting. Prefer clear thresholds that tests can assert exactly.

### Required code touch points
- `backend/src/progress/service.js`
- `backend/src/progress/service.test.js`

### Expected result
- one pure service layer defines the MVP progress heuristic clearly
- snapshot calculation is testable without DB or HTTP setup
- weak/strength area outputs are stable enough for persistence and retrieval

### Expected test outcomes
- zero-activity project yields sane zero-state snapshot
- mixed-performance topics classify weak vs strong deterministically
- borderline thresholds behave exactly as documented
- summary text stays aligned with the computed snapshot state

## D1.5.E4 — Add progress repository persistence helpers for `progress_snapshots`
- [ ] Add repository helpers to insert persisted project-level and topic-level snapshot rows
- [ ] Ensure snapshot writes are scoped to the owned project
- [ ] Add focused repository tests for persisted shape and level distinctions

### Business logic
Refreshing progress should persist a reproducible snapshot history, not just return ephemeral calculations. The system needs repository helpers that write project-level and topic-level rows into `progress_snapshots` in a consistent way.

### Exact coding problem
Implement insert helpers that persist calculated snapshot payloads for:
- one project-level snapshot
- multiple topic-level snapshots keyed by `outline_item_id`

### Required function behavior
Suggested functions:
- `createProjectSnapshot({ projectId, userId, snapshot })`
- `createTopicSnapshots({ projectId, userId, topicSnapshots })`

The persistence layer must:
1. store the correct level distinction between project and topic snapshots
2. populate required fields such as `completion_percent`, `mastery_score`, `progress_state`, `weak_areas`, `strength_areas`, and `summary_text`
3. associate topic snapshots with the correct `outline_item_id`
4. preserve snapshot history rather than mutating old rows unless the schema explicitly says otherwise

### Important persistence rule
- Do not invent destructive replacement behavior unless required by schema/spec
- If batch insert is feasible, prefer one clear batch write over many tiny inconsistent writes

### Required code touch points
- `backend/src/progress/repository.js`
- `backend/src/progress/repository.test.js`

### Expected result
- snapshot persistence helpers exist and match schema requirements
- project-level and topic-level rows are written predictably
- persisted fields are shaped for later retrieval endpoints in D1.5.F

### Expected test outcomes
- project snapshot row is inserted with the expected fields
- topic snapshot rows are inserted with the expected `outline_item_id`
- writes are rejected or blocked cleanly when ownership/project context is invalid
- multiple topic snapshots persist in a stable predictable order/shape if batching is used

- [ ] Validate `projectId` and confirm project ownership
- [ ] Read aggregates, compute snapshots, persist snapshots, and return an API-facing response
- [ ] Add focused controller tests for success, empty-data, and invalid/not-found cases

### Business logic
A project owner should be able to request a progress refresh and receive an updated persisted snapshot of overall progress plus topic-level weak/strength insights.

### Exact coding problem
Implement one controller action for:
- `POST /api/projects/:projectId/progress/refresh`

### Required controller behavior
Suggested function name:
- `refreshProjectProgress(req, res)`

The controller must:
1. validate `projectId`
2. verify the project belongs to the current user
3. call repository aggregate readers from D1.5.E2
4. call snapshot calculation service from D1.5.E3
5. persist project/topic snapshots through D1.5.E4
6. return an API-facing payload such as:
   - `{ projectId, projectSnapshot, topicSnapshots, weakAreas, strengthAreas }`

### Scope rule
- The controller should orchestrate; it should not contain all snapshot formulas inline
- Keep error handling predictable and aligned with existing project-scoped controllers

### Required code touch points
- `backend/src/progress/controller.js`
- `backend/src/progress/controller.test.js`
- possibly `backend/src/projects/repository.js` or equivalent ownership helper only if missing

### Expected result
- refresh endpoint works end-to-end at controller level
- response payload is stable enough for frontend use and later retrieval endpoints
- empty/no-attempt projects still return a sane persisted baseline snapshot rather than crashing

### Expected test outcomes
- success case persists and returns snapshot data
- invalid `projectId` is rejected
- unauthorized/not-found project is rejected cleanly
- zero-attempt project returns a valid zero-state snapshot response
