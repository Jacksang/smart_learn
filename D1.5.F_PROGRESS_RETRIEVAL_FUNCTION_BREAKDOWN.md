## D1.5.F1 — Audit current progress module and define retrieval contract
- [x] Verify the exact persisted snapshot shape and retrieval expectations
- [x] Confirm which controller/router/repository files already exist and can be extended
- [x] Document the retrieval response contract for project, topic, and weak-area endpoints
- [x] Update D1.5 notes only if the audit changes assumptions

### D1.5.F1 audit notes (2026-04-05)
- Confirmed current progress module surface already exists and is the correct extension point for D1.5.F:
  - `backend/src/progress/repository.js` currently exposes aggregate readers plus append-only snapshot writers
  - `backend/src/progress/controller.js` currently exposes only `refreshProjectProgress()` and shared `mapPersistedSnapshotToApi()`
  - `backend/src/progress/router.js` currently wires only `POST /projects/:projectId/progress/refresh`
- Confirmed persisted snapshot storage contract from `backend/db/schema/001_baseline.sql` and repository mapping:
  - table: `progress_snapshots`
  - row identity/context: `id`, `project_id`, nullable `outline_item_id`, `snapshot_type`, `created_at`
  - persisted metrics: nullable `completion_percent`, nullable `mastery_score`, nullable `progress_state`, nullable `summary_text`
  - persisted arrays/blobs: nullable JSONB `weak_areas`, nullable JSONB `strength_areas`
  - project-level snapshots are stored with `outline_item_id = NULL`
  - topic-level snapshots are stored with `outline_item_id IS NOT NULL`
  - repository/controller already normalize persisted rows to API fields via `mapProgressSnapshotRow()` -> `mapPersistedSnapshotToApi()`
- Confirmed retrieval expectation for D1.5.F: endpoints must read the latest persisted snapshot rows and must not recompute aggregates or weak areas inline. "Latest" should follow the existing append-only model and therefore sort by `created_at DESC` (with `id DESC` as a stable tiebreaker when implementing lookup helpers).
- Confirmed the refresh response created in D1.5.E is the reusable payload baseline for retrieval responses. Current API-facing snapshot shape is:
  - `id`
  - `projectId`
  - `outlineItemId`
  - `snapshotType`
  - `completionPercent`
  - `masteryScore`
  - `progressState`
  - `weakAreas`
  - `strengthAreas`
  - `summaryText`
  - `createdAt`
- Retrieval response contract for later slices in D1.5.F:
  - `GET /api/projects/:projectId/progress`
    - success body: `{ projectId, progressSnapshot }`
    - `progressSnapshot` should use the API-facing snapshot shape above and come from the newest project-level row (`outline_item_id IS NULL`)
    - if the owned project exists but no snapshot exists yet, return a clean empty success payload rather than recomputing, e.g. `{ projectId, progressSnapshot: null }`
  - `GET /api/projects/:projectId/progress/topics/:itemId`
    - success body: `{ projectId, itemId, progressSnapshot }`
    - `progressSnapshot` should use the same API-facing snapshot shape and come from the newest topic-level row for `(project_id, outline_item_id = itemId)`
    - if no topic snapshot exists yet, return `{ projectId, itemId, progressSnapshot: null }`
  - `GET /api/projects/:projectId/progress/weak-areas`
    - success body: `{ projectId, weakAreas, summaryText }`
    - source should be the newest project-level snapshot; `weakAreas` should default to `[]` when the latest snapshot stores `NULL`
    - if no project snapshot exists yet, return `{ projectId, weakAreas: [], summaryText: null }`
- Validation/ownership expectations carried forward from the current module surface:
  - validate route params before repository calls
  - enforce project ownership in repository lookup helpers, matching the existing `owned_project` pattern already used for refresh
  - keep API fields camelCase even though repository rows stay snake_case internally
- Audit conclusion: no schema/spec correction was required, so `D1.5_ANSWERS_AND_PROGRESS_SUBCHECKLIST.md` assumptions remain valid.
