## D1.5.D2 — Add repository function for project-level answer history
- [x] Add repository method for project-wide answer history
- [x] Enforce ownership through project/question joins
- [x] Support newest-first ordering and sane MVP limit behavior
- [x] Add focused repository tests

### Business logic
A project owner should be able to view recent answer activity across the whole project, not only for one question, so the app can show learning history and recent performance.

### Exact coding problem
Implement a repository function that returns answer attempts for one project owned by one user, with optional MVP limit support.

### Required function behavior
Suggested function name:
- `listRecentByProjectForUser({ projectId, userId, limit })`

The function must:
1. join through project/question ownership so only the current user's project data is returned
2. return attempts across all questions in the project
3. include enough fields for API display, such as:
   - answer attempt id
   - question id
   - attempt number
   - user answer
   - correctness
   - score
   - feedback text
   - answered at
4. order newest-first by:
   - `answered_at DESC`
   - then `created_at DESC`
   - then `id DESC` or another stable fallback
5. support a sane MVP limit default or explicit limit parameter

### Required code touch points
- `backend/src/answers/repository.js`
- `backend/src/answers/repository.test.js`

### Expected result
- one repository method exists and is ownership-safe
- query output is suitable for the API layer without extra heavy reshaping
- repository tests fully prove the data contract

### Expected test outcomes
- only matching project rows are returned
- newest-first ordering is preserved
- limit behavior works if implemented
- cross-project leakage is blocked

## D1.5.D6 — Add controller action for explicit evaluation
- [ ] Add controller for `POST /api/projects/:projectId/answers/evaluate`
- [ ] Validate `projectId` and `answerAttemptId`
- [ ] Reuse shared evaluation service from D1.5.B
- [ ] Add focused controller tests

### Business logic
A project owner should be able to ask the system to explicitly evaluate a previously submitted answer attempt and get a deterministic evaluation result without resubmitting the answer.

### Exact coding problem
Implement:
- `POST /api/projects/:projectId/answers/evaluate`

### Required controller behavior
Suggested function name:
- `evaluateProjectAnswerAttempt(req, res)`

Expected request body:
- `{ answerAttemptId }`

The controller must:
1. validate `projectId`
2. validate `answerAttemptId`
3. load the attempt + question context using D1.5.D5
4. call the shared evaluation service already created in D1.5.B
5. return an evaluation response shape such as:
   - `{ projectId, answerAttemptId, evaluation }`

### Important scope rule
- For MVP, explicit evaluation may be read-only recomputation from stored data
- Do **not** invent extra persistence unless the existing API spec clearly requires it

### Required code touch points
- `backend/src/answers/controller.js`
- `backend/src/answers/service.js` only if a tiny helper extraction is needed
- `backend/src/answers/controller.test.js`

### Expected result
- explicit evaluation works without duplicating scoring logic in the controller
- response shape is stable and API-facing

### Expected test outcomes
- success case returns evaluation payload
- invalid input is rejected
- not-found / unauthorized access is rejected cleanly
