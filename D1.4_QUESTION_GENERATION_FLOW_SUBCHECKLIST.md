## D1.4.A — Audit and realign the existing questions module to the PostgreSQL MVP schema

### Why this task exists
The current `backend/src/questions/*` code still follows an older shape (`user_id`, `outlineId`, `topic`, `type`, `difficulty`, `source`) that does not match the live PostgreSQL schema fields used by `questions` (`project_id`, `outline_item_id`, `batch_no`, `position_in_batch`, `question_type`, `difficulty_level`, `generation_source`, `status`). This mismatch should be fixed before generation logic is added.

### Deliverables
- [x] Review and document the exact runtime/schema gaps in:
  - `backend/src/questions/controller.js`
  - `backend/src/questions/repository.js`
  - `backend/src/questions/router.js`
  - `backend/db/schema/001_baseline.sql`
  - `docs/API_ENDPOINT_SPEC.md`
- [x] Refactor the questions repository shape to use schema-accurate fields and project ownership rules
- [x] Replace user-scoped list/create behavior with project-scoped helpers needed by D1.4
- [x] Save artifact: aligned repository/controller/router code + tests
- [ ] Commit and push

### D1.4.A audit notes (2026-04-04)
- Runtime gap found: `backend/src/questions/controller.js` still accepted/listed questions by `userId`, `outlineId`, and `topic`, but the PostgreSQL MVP schema defines project-owned questions with `project_id`, `outline_item_id`, `batch_no`, `position_in_batch`, `question_type`, `difficulty_level`, `generation_source`, and `status`.
- Runtime gap found: `backend/src/questions/repository.js` queried/inserted legacy columns (`user_id`, `outline_id`, `topic`, `type`, `difficulty`, `source`, `tags`) that do not exist in `backend/db/schema/001_baseline.sql`.
- Runtime gap found: `backend/src/questions/router.js` exposed top-level list/create routes, while `docs/API_ENDPOINT_SPEC.md` defines project-scoped question endpoints under `/api/projects/:projectId/questions`.
- Alignment applied: questions are now listed and fetched within project + user ownership scope via `learning_projects.user_id`, with stable ordering by `batch_no`, `position_in_batch`, and `created_at`.
- Alignment applied: project router now mounts `/:projectId/questions`, which matches the API spec and unblocks later D1.4 generation endpoints.

### D1.4.B implementation notes (2026-04-04)
- Added `backend/src/questions/service.js` with a project-scoped `loadGenerationContext()` helper and `generateQuestionBatch()` service entrypoint.
- The service now resolves the current project, current outline, and requested outline item before generation, and returns a persistence-ready question payload for the next controller step.
- MVP generation is intentionally deterministic and mock-based: prompts, options, answers, and explanations are derived from the outline path plus outline item title/content without randomness.
- Request controls now support `difficultyLevel` (`easy|medium|hard`) and `questionTypes` (`multiple_choice`, `short_answer`, `true_false`) with structured validation errors.
- Added focused unit coverage in `backend/src/questions/service.test.js` for context loading, deterministic batch generation, and invalid control validation.
