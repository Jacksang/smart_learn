## D1.1.A — Persistence strategy decision
- [x] Inspect current backend dependencies and Mongo-oriented runtime paths
  - Findings (2026-04-04): root `package.json` still depends on `mongoose`; active backend entrypoint is `backend/server.js` -> `backend/config/server.js` -> `backend/config/database.js`.
  - `backend/config/database.js` is already PostgreSQL-oriented (`pg` `Pool` + `SELECT 1` health probe), but root `package.json` does not yet declare `pg`.
  - Active runtime controllers still load Mongoose models: `backend/src/users/{controller,middleware}.js`, `backend/src/outline/controller.js`, `backend/src/questions/controller.js`, `backend/src/answers/controller.js`.
  - Mongo-only model layer remains under `backend/models/{User,Outline,Question,Answer}.js`.
- [x] Decide PostgreSQL access approach for MVP (`pg` vs ORM/query builder)
  - Decision (2026-04-04): use direct `pg` with repository modules for MVP, not an ORM/query builder.
  - Why: `backend/config/database.js` already exposes a `pg` `Pool`; new repository files under `backend/src/*/repository.js` are already following direct SQL patterns; keeping raw SQL avoids introducing a second migration variable while the team is still replacing active Mongoose runtime paths.
  - MVP boundary: prefer thin repositories per domain module and postpone ORM/query-builder adoption until after the PostgreSQL schema/bootstrap and Mongo runtime retirement are complete.
- [x] Record decision, rationale, and migration scope
  - Record (2026-04-04): Smart Learn MVP will standardize on PostgreSQL via direct `pg` access with thin per-domain repository modules, avoiding ORM/query-builder adoption during the migration foundation phase.
  - Rationale: the active backend already has a PostgreSQL-oriented pool module in `backend/config/database.js`, and repository-shaped replacements can retire Mongoose incrementally with less churn than adding an ORM at the same time.
  - Migration scope: D1.1 covers documenting the migration plan, inventorying and retiring Mongo/Mongoose runtime paths, declaring PostgreSQL dependencies/config, and adding an initial schema/bootstrap path aligned to `docs/POSTGRESQL_SCHEMA_SPEC.md`; it does not include full D1.2+ feature rewrites or real-LLM integration.
