## D2.1.1 — Audit session schema/spec/current code surface
- [x] Verify the existing schema and API expectations for learning sessions
- [x] Confirm whether any session-related backend code already exists
- [x] Define the minimum MVP session contract before implementation starts

### Business logic
Before implementing learning sessions, the system needs one clear contract for what a session is, which project it belongs to, and which state fields are required to track the learner’s current mode/topic.

### Exact coding problem
Audit current schema/docs/code so later slices can implement learning sessions without guessing about required fields, ownership rules, or response shapes.

### Required touch points
- `backend/db/schema/001_baseline.sql`
- `docs/API_ENDPOINT_SPEC.md`
- `docs/LESSON_STATE_MACHINE.md`
- `backend/src/` session-related files only if they already exist
- `TODO_DELIVERABLES_AND_CHECKPOINTS.md` only if assumptions need correction

### Audit findings
- Schema already includes `learning_sessions` with: `id`, `project_id`, `user_id`, `mode`, `status`, `current_outline_item_id`, `started_at`, `ended_at`, `session_summary`, `motivation_state`, `created_at`, `updated_at`.
- Session state machine docs define controlled mode values as: `learn`, `review`, `quiz`, `reinforce`.
- Session API spec currently describes a broader REST surface (`GET/POST /api/projects/:projectId/sessions`, `GET/PATCH /api/projects/:projectId/sessions/:sessionId`, plus `resume` and `end`).
- No dedicated backend session module exists yet under `backend/src/`.
- Existing backend surface only references sessions indirectly: answers already accept optional `sessionId` / `session_id`, and answer/progress tests include session-linked flows.

### Minimum MVP session contract confirmed for implementation
- Ownership scope: every session is project-scoped **and** user-scoped; all reads/writes must verify the project belongs to the authenticated user.
- Persistence fields required for MVP create/get/update flows:
  - `id`
  - `projectId`
  - `userId`
  - `mode`
  - `status`
  - `currentOutlineItemId`
  - `startedAt`
  - `endedAt`
  - `createdAt`
  - `updatedAt`
- Controlled enum values for MVP:
  - `mode`: `learn`, `review`, `quiz`, `reinforce`
  - `status`: at minimum `active`; later slices may also support `paused` / `completed` because both schema/docs imply resume/end behavior.
- Minimum endpoint contract needed before broader tutor-flow features:
  - `POST /api/projects/:projectId/sessions` → create/start or return the current active project session
  - `PATCH /api/projects/:projectId/sessions/:sessionId` → update `mode`, `status`, and/or `currentOutlineItemId`
  - `GET /api/projects/:projectId/sessions/:sessionId` or equivalent current-session fetch path → return persisted session state for resume
- Minimum response payload should expose stable camelCase fields:
  - `id`, `projectId`, `mode`, `status`, `currentOutlineItemId`, `startedAt`, `endedAt`, `createdAt`, `updatedAt`
- `sessionSummary` and `motivationState` exist in schema but are not required for initial MVP session-state wiring.
- Project-level `learning_projects.current_mode` should stay aligned with the active session mode when session updates are implemented.

### Expected result
- one short note/checklist update clarifies the learning-session MVP contract
- later slices know exact fields, routes, and state assumptions

### Expected test outcomes
- no new runtime tests required in this audit slice unless the audit reveals an already-broken session scaffold
