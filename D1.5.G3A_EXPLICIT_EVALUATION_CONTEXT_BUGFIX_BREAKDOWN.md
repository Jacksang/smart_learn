## D1.5.G3a.1 — Audit repository/controller shape mismatch
- [x] Confirm the exact repository return shape for explicit evaluation lookup
- [x] Confirm the exact controller field access that currently fails
- [x] Record the minimum contract that both sides must agree on

### Audit notes (2026-04-05)
- `backend/src/answers/repository.js`
  - `findAttemptWithQuestionContextForProjectAndUser(...)` returns `mapAnswerAttemptWithQuestionContextRow(...)`.
  - That mapper returns the persisted attempt fields at the top level (`id`, `project_id`, `user_answer`, etc.) and nests question context under `answerAttempt.question`.
  - The nested `question` object currently contains:
    - `id`
    - `prompt`
    - `question_type`
    - `correct_answer`
    - `explanation`
    - `outline_item_id`
- `backend/src/answers/controller.js`
  - `evaluateProjectAnswers` currently calls `evaluateAnswerAttempt(...)` with:
    - `question.question_type = answerAttempt.question_type`
    - `question.correct_answer = answerAttempt.correct_answer`
    - `question.explanation = answerAttempt.explanation`
  - Those reads expect explicit-evaluation question context at the top level of `answerAttempt`, not under `answerAttempt.question`.
- Confirmed mismatch:
  - repository contract now: `answerAttempt.question.question_type` / `.correct_answer` / `.explanation`
  - controller expectation now: `answerAttempt.question_type` / `.correct_answer` / `.explanation`
  - Result: explicit evaluation loses the stored question context even when the lookup succeeds.

### Minimum corrected contract
- The repository/controller boundary must expose one consistent source of question context for explicit evaluation.
- Minimum required fields for evaluation are:
  - persisted answer attempt top-level: `id`, `project_id`, `user_answer`
  - question evaluation context available in one agreed location: `question_type`, `correct_answer`, `explanation`
- Minimal safe alignment options for D1.5.G3a.2:
  1. keep repository shape nested and make controller read `answerAttempt.question.question_type`, `answerAttempt.question.correct_answer`, and `answerAttempt.question.explanation`, or
  2. flatten those three question fields onto the repository return shape so the existing controller reads stay valid.
- Smallest contract to preserve going forward:
  - if nested shape is kept, controller tests should explicitly lock `answerAttempt.question.*` as the evaluation source.
  - if flat shape is chosen instead, repository tests should explicitly lock those fields at top level and avoid ambiguous dual-shape access.

### Business logic
Explicit evaluation should read one stored answer attempt and its related question context in one consistent shape, so the controller can deterministically evaluate a past answer without resubmission.

Audit the current repository/controller mismatch and define the corrected contract before implementing the fix.
